全面梳理和重新识别社区老年助餐服务系统项目的完整状态，建立准确的项目认知基线，为后续论文编写和功能修改提供可靠依据。

## 二、项目概况

### 2.1 项目定位

本项目 `Csma`（社区老年助餐服务系统）基于原始"苍穹外卖"项目改造而来，从商用外卖后台调整为社区老年助餐服务平台。系统服务于社区老年人订餐配送场景。

### 2.2 技术栈

| 层级    | 技术                                       | 版本                     | 说明                                            |
| ----- | ---------------------------------------- | ---------------------- | --------------------------------------------- |
| 前端    | Vue 2.6 + TypeScript 3.9 + Element UI    | Vue 2.6.14, TS 3.9     | 管理后台与家属端共享同一工程 `sky-admin-vue`                |

## 四、核心角色与权限体系

### 4.1 角色定义

| 角色          | 来源表        | 路由入口                  | 职责                    |
| ----------- | ---------- | --------------------- | --------------------- |
| `ADMIN`     | `employee` | `/dashboard`          | 全局管理：订单调度、基础数据、员工管理   |
| `OPERATOR`  | `employee` | `/order`              | 助餐点操作：处理本点订单、出餐、分配志愿者 |
| `FAMILY`    | `user`     | `/family-order`       | 家属端：为老人点餐、查看历史订单、评价   |
| `VOLUNTEER` | `user`     | `/volunteer-overview` | 志愿者：取餐配送、查看个人概览与任务    |

### 4.2 权限控制方式（三层管控）

#### 后端 — 拦截器级

* `JwtTokenAdminInterceptor`：拦截 `/admin/**`，排除 `/admin/employee/login`

  * 从 JWT 提取 `empId`、`ROLE`、`DINING_POINT_ID` 注入 `BaseContext`

  * OPERATOR 只有 `/admin/order/` 和 `/admin/shop/` 两个路径前缀权限，其他返回 403

  * ADMIN 拥有全部 `/admin/**` 权限

  * 使用 `adminSecretKey` 验证 Token

* `JwtTokenUserInterceptor`：拦截 `/user/**`，排除 `/user/user/login`、`/user/user/register`、`/user/shop/status`

  * 从 JWT 提取 `userId`、`ROLE`（默认 FAMILY），`DINING_POINT_ID` 强制设为 null

  * 使用 `userSecretKey` 验证 Token（与 Admin 不同密钥）

* Token 校验后统一通过 `afterCompletion` 清理 `BaseContext`

#### 后端 — 服务层

* `OrderServiceImpl.requireRole()` 在操作入口做角色校验

* `OrderServiceImpl.validateDataScope()` / `applyQueryScope()` 按角色做数据隔离：

  * OPERATOR 按 `diningPointId` 过滤

  * VOLUNTEER 按 `volunteerId` 过滤

  * FAMILY 按 `userId` 过滤

#### 前端 — 路由级

* `router.ts` 中 `meta.roles: ['ADMIN']` 控制页面访问

* `permission.ts` beforeEach 守卫：

  * 有 token 但 role 不合法 → 清除 Cookie，跳转 `/login`

  * 已登录访问 `/login` → 按角色跳转默认页

  * 目标路由有 roles 限制 → 检查当前角色是否在允许列表中

  * 无 token → 仅允许 `notNeedAuth: true` 的页面

* 登录态通过 Cookie 持久化（`js-cookie`），Vuex 作为运行时缓存

## 五、核心业务规则

### 5.1 业务主线

```
FAMILY 选择老人 → 加载老人助餐点菜品 → 加入购物车 → 提交订单
→ 支付（15分钟超时）→ 订单进入调度 → OPERATOR 处理
→ 分配志愿者 → 出餐 → VOLUNTEER 取餐配送 → 送达完成 → FAMILY 评价
```

### 5.2 关键规则

1. **老人驱动订单归属**：`elderly.dining_point_id` → `orders.dining_point_id`，菜品只做合法性校验
2. **购物车绑定老人**：同一 FAMILY 用户同一时刻只允许一个老人的购物车，切换老人清空购物车。支持菜品和套餐双条目类型
3. **助餐点休息态**（`dining_point.status=0`）：阻断新单（禁止加购/下单/支付），不影响已有订单继续履约
4. **订单状态流转**（`orders.status`）：

   * `1` 待支付 → `2` 待调度 → `3` 制作中 → `4` 待取餐 → `5` 配送中 → `6` 已完成

   * 任意状态 → `7` 已取消
5. **定时任务**（Spring `@Scheduled`）：

   * 支付超时自动取消：待支付订单超过 15 分钟自动取消

   * 逾期订单日志识别：每 10 分钟扫描异常履约订单，仅日志不修改状态
6. **FAMILY 金额模型**：`deliveryFee=0`、`tablewareFee=1元/份`、`subsidyAmount=0`、`payAmount=dishAmount+tablewareFee`
7. **金额防篡改**：`submitOrder` 中服务端独立计算金额并与前端提交金额逐项比对
8. **评价体系**：一单一评、1\~5 分、仅 FAMILY 评价已完成订单、评分聚合计入志愿者综合评分
9. **催单限流**：ConcurrentHashMap 实现 60 秒催单频率限制
10. **WebSocket 实时通知**：支付成功后推送给对应助餐点操作员（type=1 来单提醒），催单推送 type=2

### 5.3 志愿者等级规则（VolunteerLevelSupport.calculateLevel）

| 等级      | 累计服务单量     |
| ------- | ---------- |
| Level 1 | 0 \~ 9 单   |
| Level 2 | 10 \~ 29 单 |
| Level 3 | 30 \~ 59 单 |
| Level 4 | 60 \~ 99 单 |
| Level 5 | 100 单及以上   |

每次评价后通过 `refreshVolunteerRating()` 重新计算。

## 七、前端关键实现细节

### 7.1 登录认证流

```
/login 页面 → loginType 参数区分 admin/user
  → admin: POST /admin/employee/login
  → user:  POST /user/user/login
  → 后端返回 JWT token + userInfo
  → JWT decode 提取 role
  → 存储到 Cookie: token, role, user_info
  → Vuex 同步更新 user 模块
  → 按角色跳转对应首页
```

**重要**：前端依赖 JWT decode 提取 role，这意味着 JWT payload 中必须包含 `role` 字段。

### 7.2 请求代理规则

```javascript
// vue.config.js devServer.proxy
// 规则 1（优先匹配）: /api/user → http://localhost:8080 （去 /api 前缀）
'/api/user': { target: 'http://localhost:8080', pathRewrite: { '^/api': '' } }

// 规则 2: /api → http://localhost:8080/admin （去 /api 前缀）
'/api': { target: 'http://localhost:8080/admin', pathRewrite: { '^/api': '' } }
```

生产环境通过 Nginx 反向代理实现同等效果。

### 7.3 请求去重机制

Axios 请求拦截器中基于 `md5(url + method + JSON.stringify(data))` 生成请求 key，检查 `pending` 字典，重复请求直接 cancel。响应拦截器和错误处理中从 `pending` 字典移除 key。

### 7.4 金额精度处理（`familyCart.ts`）

所有金额计算使用\*\*分（cent）\*\*为单位的整数运算：

* `toCent(amount)` — 元 → 分（`Math.round(amount * 100)`）

* `fromCent(amount)` — 分 → 元（`(amount / 100)`）

* `formatAmount(amount)` — 格式化显示 `amount.toFixed(2)`

### 7.5 家属点餐页面核心组件

`user-order/index.vue`（2350 行）是最大的单页面组件，支持**分类/套餐双模式切换**，核心 UI 结构：

```
├── 顶部：模式切换 Tab（分类 | 套餐）+ 助餐点名称 + 营业状态标识
├── 左侧面板（category-panel）
│   ├── 分类模式：菜品分类列表 → 菜品网格（DishCard）
│   └── 套餐模式：套餐分类列表 → 套餐网格（SetmealCard）
│       └── SetmealCard 点击 → 套餐详情弹窗（图片/描述/价格/套餐内菜品列表/加入购物车）

├── 底部：固定购物车栏（cartBar）
│   ├── 购物车图标 + 商品数量

└── 结算弹窗：选择老人 → 确认菜品/套餐 → 地址 → 备注 → 提交
```

### 7.6 状态管理风格不一致

## 十一、架构总结与设计特点

### 技术特点

1. **双密钥 JWT**：Admin 端和 User 端使用不同的 JWT 密钥，安全隔离
2. **ThreadLocal 上下文传递**：`BaseContext` 避免显式参数传递
3. **混合 SQL 风格**：简单查询用注解，复杂动态查询用 XML
4. **模拟支付**：当前支付为模拟实现，`paySuccess` 需手动触发
5. **状态 / 业务校验分离**：状态校验用 `assertStatus()`，权限用 `requireRole()`，数据范围用 `validateDataScope()`

### 代码量分布

| 文件                      | 行数   | 说明       |
| ----------------------- | ---- | -------- |
| `OrderServiceImpl.java` | 1552 | 最复杂的服务实现 |
| `user-order/index.vue`  | 2350 | 最复杂的前端页面 |
| `db_design.md`          | 425  | 数据库完整设计  |
| `spec.md`               | 821  | 项目规格说明   |

### 前端注意点

* Vue 2 + class-based 组件风格，不要使用 Composition API

* Element UI（非 Element Plus）

* SCSS 全局变量通过 `vue.config.js` 自动注入

* `store/modules/cart.ts` 使用 Pinia，其他模块使用 Vuex，新增功能建议统一

### 后端注意点

* 密码使用 MD5（历史兼容，不要求 BCrypt）
* MyBatis XML 为主，注解 SQL 为辅
* `SetmealMapper` 同时被 `ShoppingCartServiceImpl` 和 `OrderServiceImpl` 注入使用
* ShoppingCart 中 `setmealId` 与 `dishId` 互斥——一条购物车记录要么是菜品，要么是套餐
* 套餐金额在 `addShoppingCart` 时从 `Setmeal.price` 写入 `ShoppingCart.amount`，后续结算、下单直接读取 `amount` 字段，无需区分类型
* 字段命名：Java 侧 `camelCase`，数据库 `snake_case`（由 MyBatis `map-underscore-to-camel-case` 自动转换）

* 统一返回 `Result<T>`，分页用 `PageResult`

### 数据隔离

* **OPERATOR** 只能看到所属 `dining_point_id` 的订单

* **VOLUNTEER** 只能看到分配给自己的已完成订单

* **FAMILY** 只能看到自己的订单

* **ADMIN** 无数据限制，可看到全部
