# 社区老年助餐服务系统 - 项目规格说明书 (Spec)

## 1. 项目概述
本项目旨在改造现有的 **苍穹外卖 (Sky Take Out)** 系统，转型为 **社区老年助餐服务平台**。核心业务模式从商业外卖转变为公益性质的社区助餐服务，强调“子女代订、志愿者派送、社区统一调度”。

- **目标用户**: 社区老人及其家属、社区志愿者、助餐点工作人员、社区管理员。
- **核心价值**: 解决独居/高龄老人用餐难问题，整合社区资源提供精准服务。

## 2. 技术架构
### 2.1 后端 (Spring Boot)
- **基础框架**: Spring Boot 2.x
- **持久层**: MyBatis + MySQL 5.7+
- **缓存**: Redis (用于缓存菜品、用户Token)
- **安全**: Spring Security + JWT (多角色鉴权)
- **工具**: Lombok, Swagger/Knife4j, POI (报表导出)

### 2.2 前端 (Vue 2 + Element UI)
- **架构**: 单页应用 (SPA)，基于 `vue-element-admin` 及其简化版改造。
- **UI库**: Element UI (PC端 Web)。
- **适老化**: 针对老人/家属端进行大字体、高对比度适配。
- **动态路由**: 根据登录角色 (Admin/Operator/Volunteer/Family) 动态加载菜单和路由。

## 3. 角色与功能模块

### 3.1 社区管理员 (Admin)
- **入口**: PC Web 后台。
- **权限**: 最高权限，管理所有资源。
- **功能**:
  - **助餐点管理**: CRUD (名称、地址、电话、营业状态)。
  - **餐品管理**: 维护菜品信息，关联助餐点，设置营养标签 (低糖/低盐/软食)。
  - **人员管理**: 审核志愿者注册，管理老人档案，管理操作员账号。
  - **订单调度**: 查看待调度订单，手动指派志愿者 (基于片区/空闲状态)。
  - **数据统计**: 订单量、志愿者工时、补贴金额统计。

### 3.2 助餐点操作员 (Operator)
- **入口**: PC Web (简化版后台)。
- **权限**: 仅限所属助餐点数据。
- **功能**:
  - **今日订单**: 查看分配到本点的订单 (待制作/制作中)。
  - **出餐管理**: 标记订单为“待取餐”，打印小票。
  - **核销交接**: 确认志愿者取餐 (可选扫码/点击确认)。

### 3.3 志愿者 (Volunteer)
- **入口**: PC Web (任务看板)。
- **权限**: 仅限查看分配给自己的任务。
- **功能**:
  - **任务列表**: 待取餐、配送中、已完成任务。
  - **任务详情**: 查看取餐点地址、送餐老人地址、特殊需求备注。
  - **状态更新**: 点击“确认取餐”、“确认送达”。
  - **个人中心**: 查看服务时长、积分/荣誉。

### 3.4 老人/家属 (Family/Elderly)
- **入口**: PC Web (C端门户)。
- **权限**: 普通用户权限。
- **功能**:
  - **注册/登录**: 手机号/账号登录。
  - **档案管理**: 绑定老人信息 (多位)，填写健康状况、忌口。
  - **点餐下单**: 浏览菜品 (按营养/口味筛选)，选择就餐老人，预约送达时间。
  - **模拟支付**: 展示“政府补贴 -XX元，自付 XX元”，点击支付完成。
  - **订单中心**: 追踪订单状态，评价服务。

## 4. 核心业务流程

### 4.1 订餐流程
1. **下单**: 家属登录 -> 选择菜品 -> 选择老人 -> 确认订单 (系统计算补贴) -> 支付 (模拟)。
2. **状态流转**: 订单状态变为 `TO_BE_SCHEDULED` (待调度)。

### 4.2 调度与制作流程
1. **调度**: 管理员后台看到 `TO_BE_SCHEDULED` 订单 -> 选择志愿者 -> 派单。
   - 订单状态 -> `CONFIRMED` (已接单/制作中)。
   - 志愿者端看到新任务。
   - 助餐点端看到新订单。
2. **制作**: 助餐点完成制作 -> 点击“出餐”。
   - 订单状态 -> `MEAL_READY` (待取餐)。

### 4.3 配送流程
1. **取餐**: 志愿者到达助餐点 -> 核对信息 -> 确认取餐。
   - 订单状态 -> `DELIVERING` (配送中)。
2. **送达**: 志愿者送餐上门 -> 确认送达。
   - 订单状态 -> `COMPLETED` (已完成)。
   - 系统记录志愿者服务时长。

## 5. 开发任务规划 (Tasks)

### Phase 1: 数据库与后端基础改造
1. **数据库迁移**: 执行 `db_design.md` 中的 SQL 变更。
2. **实体类更新**: 修改 `User`, `Employee`, `Order` 等 POJO，新增 `Elderly`, `DiningPoint` 等实体。
3. **Mapper/DAO**: 编写新表的 CRUD 接口。

### Phase 2: 后端业务逻辑实现
1. **用户模块**: 实现多角色登录鉴权 (Spring Security 配置调整)。
2. **管理端接口**: 助餐点管理、老人档案管理、调度派单接口。
3. **C端接口**: 菜品浏览 (带营养标签)、下单 (带补贴逻辑)、模拟支付。
4. **作业端接口**: 志愿者任务列表、状态更新 (取餐/送达)。

### Phase 3: 前端页面开发 (Vue 2)
1. **框架调整**: 引入动态路由，根据角色加载不同 Layout。
2. **公共组件**: 封装适老化 Text 组件、高对比度主题。
3. **管理端页面**: 助餐点CRUD、人员管理、调度中心。
4. **C端页面**: 首页、点餐页、订单页、个人中心。
5. **作业端页面**: 志愿者任务看板、操作员订单处理页。

### Phase 4: 联调与测试
1. **全流程测试**: 模拟从下单到送达的完整闭环。
2. **权限测试**: 确保各角色无法越权访问。
3. **适老化验收**: 检查字体大小、对比度。

## 6. 注意事项
- **数据安全**: 老人隐私信息 (住址、电话) 需脱敏展示。
- **并发控制**: 虽然业务量不大，但需防止订单状态并发修改 (如管理员和志愿者同时操作)。
- **扩展性**: 预留对接真实支付和物联网设备 (智能手环) 的接口。
