# 老人作为连接点规则说明

本文件是当前业务规则的单点真相。

## 1. 一句话定义

`老人决定订单归属助餐点，助餐点负责履约，菜品助餐点只做合法性校验。`

## 2. 业务链路

当前主链路如下：

`FAMILY 下单 -> 老人 -> 默认助餐点 -> 订单 -> OPERATOR 处理 -> VOLUNTEER 配送`

## 3. 关键规则

### 3.1 老人

- 每位老人绑定一个默认助餐点
- 关键字段：`elderly.dining_point_id`

### 3.2 订单

- 订单 `diningPointId` 直接继承自老人
- 关键字段：
  - `orders.elder_id`
  - `orders.dining_point_id`

### 3.3 菜品

- 菜品仍然绑定助餐点
- 关键字段：`dish.dining_point_id`
- 当前用途：校验菜品是否合法属于当前老人对应助餐点

### 3.4 购物车

- 购物车显式绑定老人
- 关键字段：`shopping_cart.elder_id`
- 同一 FAMILY 用户同一时刻只允许保留一个老人的购物车

## 4. 前端执行规则

- FAMILY 点餐页必须先选择老人
- 菜品列表按老人绑定助餐点加载
- 切换老人时，如购物车非空，必须先清空购物车
- 结算页中的老人信息只读展示
- 提交订单时固定使用当前页面选中的老人

## 5. 后端执行规则

- `submitOrder` 不再从菜品集合推导订单助餐点
- 后端必须强校验：
  - `elderId` 非空
  - 老人存在
  - 老人属于当前 FAMILY
  - 老人已绑定助餐点
  - 购物车项属于该老人
  - 菜品属于该老人助餐点

## 6. 助餐点的真实角色

助餐点不是附属标签，而是订单履约主体。

这意味着：

- 订单落到哪个助餐点，由老人决定
- `OPERATOR` 只处理自己助餐点的订单
- 志愿者后续从该助餐点取餐并配送

## 7. 管理员维护要求

为了让“老人决定订单归属助餐点”规则稳定成立，管理员端老人档案必须承担所属助餐点的维护职责。

- 管理员在老人档案中使用统一字段：`所属助餐点`
- 该字段对应数据库：`elderly.dining_point_id`
- 新增老人时必须绑定所属助餐点
- 编辑老人时必须保留或改绑到一个有效助餐点
- 只能绑定到存在且启用中的助餐点
- 若提交不存在或已停用的助餐点，后端应明确拒绝保存

管理端当前展示约定：

- 老人列表页直接显示 `diningPointName`
- 历史空值数据展示为 `未绑定`
- 但空值老人不满足后续下单前置条件
