# 管理员页面状态字段分析与优化计划

## 一、用户问题

> "管理员页面是不是没必要显示营业状态之类的"

需要分析 ADMIN（管理员）端各页面展示的状态字段，判断哪些是必要的、哪些是不必要的冗余展示。

---

## 二、现状分析

### 2.1 ADMIN 端页面及展示的状态字段

| 页面 | 路由 | 展示的状态字段 |
|------|------|--------------|
| 仪表盘 (Dashboard) | `/dashboard` | 订单状态统计卡片：待调度/制作中/已完成/已取消 |
| 订单管理 | `/order` | 订单状态筛选：全部/待调度/制作中/待取餐/配送中/已完成/已取消；订单列表行展示状态标签 |
| 助餐点管理 | `/diningPoint` | 助餐点营业状态：营业中(1)/休息中(0)，列表+筛选 |
| 菜品管理 | `/dish` | 菜品状态：起售(1)/停售(0) |
| 老人档案 | `/elderly` | 老人状态：正常(1)/停用(0) |
| 家属档案 | `/familyProfile` | 家属档案状态：启用(1)/停用(0) |
| 员工管理 | `/employee` | 员工状态：启用(1)/停用(0)+角色 |

### 2.2 ORDER 端页面展示的状态字段

| 页面 | 路由 | 额外展示的状态字段 |
|------|------|--------------|
| 订单中心 | `/order` | 交接状态（未分配/待出餐/待交接/已取餐）、期望送达、送达时间 |

### 2.3 对比发现

**ADMIN 的订单详情**比 OPERATOR 少了6个字段：
- 老人电话、志愿者电话、交接状态、期望送达、送达时间、特殊需求

**ADMIN 多了 OPERATOR 没有的页面**：
- 仪表盘（Dashboard）—— 但仅展示4个状态统计卡片
- 助餐点管理、菜品管理、老人档案、家属档案、员工管理

### 2.4 当前冗余字段（VO中返回但页面从未展示）

以下字段存在于 `OrderVO` 返回体中，但 ADMIN 和 OPERATOR 页面均不渲染：

| 字段 | 含义 | 问题 |
|------|------|------|
| `payStatus` | 支付状态 0/1/2 | ADMIN/OPERATOR 不需要关注支付状态 |
| `deliveryStatus` | 配送方式标识 | 当前固定为0，无展示价值 |
| `payMethod` | 支付方式 | 仅模拟支付链路使用 |
| `personalPay` | 自付金额 | 从未在 ADMIN/OPERATOR 页面渲染 |
| `packAmount` | 打包费 | 当前固定为0 |
| `userName` | 下单用户名 | 页面用老人姓名取代 |
| `subsidyAmount` | 补贴金额 | 固定为0，仅在 FAMILY 端展示 |

### 2.5 历史包袱字段名

| VO字段名 | 字面含义 | 实际语义 | 问题 |
|---------|---------|---------|------|
| `OrderOverViewVO.deliveredOrders` | "已派送" | 制作中 (status=3) | 字段名误导 |
| `OrderStatisticsVO.deliveryInProgress` | "配送中" | 待取餐 (status=4) | 字段名与实际不符 |
| `orders.confirmed` (status=3) | —— | 制作中 | spec.md 中状态语义映射需确认 |

## 三、分析结论

### 3.1 必要的状态展示（建议保留）

| 页面 | 状态字段 | 保留理由 |
|------|---------|---------|
| 订单管理 | 订单状态 (1-7) | 管理员需要查看全量订单流转状态，进行调度决策 |
| 助餐点管理 | 营业状态 | 管理员需要管理助餐点开关，控制新单入口 |
| 菜品管理 | 起售/停售 | 管理员需要维护菜品有效性 |
| 老人/家属/员工 | 启用/停用 | 管理员需要维护基础数据有效性 |
| 仪表盘 | 待调度/制作中/已完成/已取消 | 管理员需要全局运营概览 |

### 3.2 可精简的状态展示

| 页面 | 可精简项 | 理由 |
|------|---------|------|
| 仪表盘 | 订单统计卡片从4个精简 | 仪表盘是聚合视图，"待取餐"和"配送中"对管理员决策价值低，可合并或移除 |
| 订单列表 | 状态筛选从7个合并 | "待取餐"和"配送中"更偏向 OPERATOR 的执行视图，ADMIN 可合并为"履约中" |

### 3.3 ADMIN 与 OPERATOR 的职责边界

| 角色 | 应关注的状态 | 不应关注 |
|------|-----------|---------|
| **ADMIN** | 全局运营（待调度量、完成量、取消量、助餐点营业状态、基础数据有效性） | 交接状态、配送进度、志愿者状态、支付状态 |
| **OPERATOR** | 本点履约（制作中、待交接、配送中、交接状态、期望送达） | 全局统计、其他助餐点订单、支付状态、基础数据管理 |

## 四、建议修改方案

### 方案 A：最小改动（仅清理冗余展示）

1. **仪表盘订单统计**：从当前5项（全部+待调度+制作中+已完成+已取消）保持不动，这已经是合理的 MANAGER 视图
2. **订单列表筛选**：保持7状态筛选不变，但可考虑折叠"待取餐"和"配送中"为可展开的子状态
3. **不额外添加**：不在 ADMIN 页面添加交接状态、配送进度等 OPERATOR 专属字段

### 方案 B：职责清晰化（推荐）

1. **ADMIN 仪表盘**：保持当前状态统计不变（待调度/制作中/已完成/已取消），这是合理的管理视图
2. **ADMIN 订单列表**：保持7状态筛选，但表格列移除过于细节的状态标签颜色区分，统一为一个简洁的状态列
3. **不在 ADMIN 页面展示**：交接状态、期望送达、送达时间等 OPERATOR 执行层字段
4. **不在 OPERATOR 页面展示**：助餐点营业状态（管理员才能改）、其他助餐点订单

### 方案 C：用户自定义

根据您的具体需求，选择性调整某个页面或某个字段的展示。

---

## 五、执行计划

### 步骤 1：确认范围
- 确定用户想调整的页面范围（仪表盘？订单管理？助餐点管理？全部？）
- 确定"营业状态"具体指什么（助餐点营业状态？订单各环节状态？）

### 步骤 2：按页面逐项调整
- 根据用户选择的方案，逐页面调整状态字段的展示
- 前端优先改动（隐藏/合并展示列）
- 后端接口可保持不变（VO 返回多字段不影响性能）

### 步骤 3：验证
- 确认 ADMIN 登录后各页面展示正常
- 确认 OPERATOR 登录后各页面展示正常
- 确认两角色之间无越权信息泄露
