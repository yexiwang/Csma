# 老人作为连接点联调测试说明

本文档用于验证当前“老人作为连接点”方案是否已在前后端链路中稳定落地。

## 1. 测试前提

- 已执行最新数据库脚本
- 已准备可登录账号：
  - `FAMILY`
  - `OPERATOR`
  - `VOLUNTEER`
- 测试老人至少包含两种情况：
  - 已绑定助餐点的老人
  - 未绑定助餐点的老人
- 测试菜品至少包含两种情况：
  - 属于老人对应助餐点的菜品
  - 不属于老人对应助餐点的菜品

## 2. 正常主链路

### 2.1 老人已绑定助餐点，正常下单

1. 使用 `FAMILY` 登录 `/family-order`
2. 选择一个已绑定助餐点的老人
3. 确认页面只展示该助餐点的菜品
4. 加购商品并进入结算
5. 确认结算页展示：
   - 当前老人
   - 当前助餐点
   - 配送地址
   - 菜品明细
   - 金额汇总
6. 提交订单并完成支付

预期：

- 请求体中包含正确的 `elderId`
- 订单创建成功
- 数据库中 `orders.dining_point_id = elderly.dining_point_id`

### 2.2 对应操作员可见

1. 使用该老人对应助餐点的 `OPERATOR` 登录
2. 进入 `/order`
3. 在本站点看板中查看订单

预期：

- 操作员能看到该订单
- 订单只出现在对应助餐点，不会出现在其他助餐点操作员视图中

### 2.3 志愿者后续履约

1. 在 `OPERATOR` 端为订单分配志愿者
2. 标记出餐完成
3. 由 `OPERATOR` 或 `VOLUNTEER` 确认取餐
4. 使用 `VOLUNTEER` 登录查看任务并完成送达

预期：

- 订单状态按链路正常流转
- 志愿者处理的是该助餐点订单

## 3. 异常场景

### 3.1 FAMILY 无关联老人

1. 使用没有关联老人的 `FAMILY` 登录 `/family-order`

预期：

- 页面只提示“请先新增或关联老人档案”
- 不展示可下单菜品
- 不允许加购和结算

### 3.2 老人未绑定助餐点

1. 选择一个未绑定助餐点的老人

预期：

- 页面提示“当前老人未绑定助餐点，无法下单”
- 不展示可购买菜品
- 后端也拒绝加购和提交订单

### 3.3 切换老人后购物车清空

1. 选择老人 A 并加购商品
2. 切换到老人 B

预期：

- 页面先弹出确认
- 确认后调用清空购物车接口
- 清空成功后才切换老人
- 取消后保持原老人和原购物车

### 3.4 不同助餐点菜品不能混下单

1. 构造购物车中包含不属于当前老人助餐点的菜品
2. 提交订单

预期：

- 前端应阻止继续结算或进入异常态
- 后端必须拒绝提交
- 返回明确错误：所选菜品不属于当前老人对应助餐点

### 3.5 提交订单时 `elderId` 缺失

1. 在前端或抓包层移除 `elderId`
2. 提交订单

预期：

- 前端先拦截并提示
- 若绕过前端，后端返回明确错误

## 4. 历史兼容

### 4.1 历史订单展示

1. 打开历史订单页或后台订单详情
2. 检查缺失老人/助餐点字段的旧订单

预期：

- 页面可正常打开
- 缺失信息显示为：
  - `历史订单未记录服务老人`
  - `历史订单未记录助餐点`

### 4.2 历史购物车脏数据

1. 构造 `shopping_cart.elder_id` 为空的历史数据
2. 打开 `/family-order`

预期：

- 页面进入异常阻塞态
- 明确提示需要先清空购物车
- 不允许继续下单

## 5. 当前主规则复核

本轮联调通过的标志是：

- 订单助餐点由老人决定
- 菜品助餐点只做合法性校验
- 购物车严格绑定老人
- 助餐点作为订单履约主体生效
