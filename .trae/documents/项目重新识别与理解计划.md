# 项目重新识别与理解计划

## 一、目标

全面梳理和重新识别社区老年助餐服务系统项目的完整状态，建立准确的项目认知基线，为后续论文编写和功能修改提供可靠依据。

---

## 二、项目概况

### 2.1 项目定位

本项目 `Csma`（社区老年助餐服务系统）基于原始"苍穹外卖"项目改造而来，从商用外卖后台调整为社区老年助餐服务平台。系统服务于社区老年人订餐配送场景。

### 2.2 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 前端 | Vue 2.6 + TypeScript 3.9 + Element UI | 管理后台与家属端共享同一工程 `sky-admin-vue` |
| 后端 | Spring Boot + MyBatis XML + Maven | 分层架构 `sky-common` / `sky-pojo` / `sky-server` |
| 数据库 | MySQL | 通过 MyBatis XML 访问，字段 `snake_case` |
| 缓存 | Redis | 可选组件 |
| 认证 | JWT + Interceptor + BaseContext | 无状态认证，区分员工端和用户端两套拦截器 |

## 四、核心角色与权限体系

### 4.1 角色定义

| 角色 | 来源表 | 路由入口 | 职责 |
|------|--------|----------|------|
| `ADMIN` | `employee` | `/dashboard` | 全局管理：订单调度、基础数据、员工管理 |
| `OPERATOR` | `employee` | `/order` | 助餐点操作：处理本点订单、出餐、分配志愿者 |
| `FAMILY` | `user` | `/family-order` | 家属端：为老人点餐、查看历史订单、评价 |
| `VOLUNTEER` | `user` | `/volunteer-overview` | 志愿者：取餐配送、查看个人概览与任务 |

### 4.2 权限控制方式

- **后端**：`JwtTokenAdminInterceptor`（员工端） + `JwtTokenUserInterceptor`（用户端），URI级控制
- **前端**：路由 `meta.roles` 控制菜单与页面访问
- **登录态**：Cookie存储 `token`/`role`/`user_info`，JWT无状态认证
- **密码**：MD5加密（当前兼容方案，不要求BCrypt）

## 五、核心业务规则

### 5.1 业务主线

```
FAMILY 下单 → 老人决定助餐点 → 订单进入调度 → OPERATOR 处理 → VOLUNTEER 配送 → 完成
```

### 5.2 关键规则

1. **老人驱动订单归属**：`elderly.dining_point_id` → `orders.dining_point_id`，菜品只做合法性校验
2. **购物车绑定老人**：同一FAMILY用户同一时刻只允许一个老人的购物车
3. **助餐点休息态**：阻断新单（禁止加购/下单/支付），不影响已有订单继续履约
4. **订单状态流转**：1待支付→2待调度→3制作中→4待取餐→5配送中→6已完成（或7已取消）
5. **定时任务**：仅保留支付超时自动取消（15分钟）+ 逾期订单日志识别（不改状态）
6. **FAMILY金额模型**：`deliveryFee=0`、`tablewareFee=1元/份`、`subsidyAmount=0`、`payAmount=dishAmount+tablewareFee`
7. **评价体系**：一单一评、1~5分、仅FAMILY评价已完成订单、评分聚合计入志愿者综合评分

### 5.3 志愿者等级规则

| 等级 | 累计服务单量 |
|------|-------------|
| Level 1 | 0 ~ 9 单 |
| Level 2 | 10 ~ 29 单 |
| Level 3 | 30 ~ 59 单 |
| Level 4 | 60 ~ 99 单 |
| Level 5 | 100 单及以上 |

## 六、数据库核心实体

| 表名 | 说明 | 核心字段 |
|------|------|----------|
| `user` | 用户表（FAMILY/VOLUNTEER） | id, username, password(MD5), role, status, name, phone |
| `employee` | 员工表（ADMIN/OPERATOR） | id, username, password(MD5), role, dining_point_id, status |
| `elderly` | 老人档案 | id, user_id(FK→user), dining_point_id(FK→dining_point), name, phone, address |
| `family_profile` | 家属档案 | id, user_id(FK→user), remark, status（name/phone来自user表） |
| `dining_point` | 助餐点 | id, name, address, status(1营业/0休息) |
| `dish` | 菜品 | id, category_id, dining_point_id, name, price, status |
| `shopping_cart` | 购物车 | id, user_id, elder_id, dish_id, number |
| `orders` | 订单 | id, number, user_id, elder_id, volunteer_id, dining_point_id, status(1-7), amount |
| `order_detail` | 订单明细 | id, order_id, dish_id, name, number, price |
| `order_review` | 订单评价 | id, order_id(唯一), score(1-5), content |
| `volunteer_stats` | 志愿者统计 | user_id, total_orders, total_hours, rating, level |

## 九、后续验证规范

在生成任何流程图或代码内容时，必须遵循：

1. **生成前验证**：先通过搜索代码库、查询规则文档确认功能的实际业务流程
2. **生成后复核**：完成后对照真实代码和业务规则交叉核对
3. **真实性原则**：不添加项目中不存在的功能点（如库存校验、自动完成配送等）
4. **代码以Service层为主**：核心代码展示以Service层业务逻辑为主
5. **流程图样式统一**：开始/结束用圆角矩形、操作步骤用矩形、判断条件用菱形、分支标注"是/否"
